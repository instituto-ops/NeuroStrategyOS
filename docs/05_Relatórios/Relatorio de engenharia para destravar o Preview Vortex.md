Esse artefato deve ser uma árvore de arquivos ou um payload estruturado de componentes, e não uma mistura oportunista de JSX, HTML paralelo e convenções implícitas de CDN. ## O que realmente está quebrando

O primeiro problema é **mismatch de runtime**. O ecossistema Next.js não é apenas “JSX no navegador”; ele pressupõe compilação, transformação e, em muitos casos, capacidades de servidor. A documentação oficial do framework deixa claro que o Next.js configura compiladores e bundlers automaticamente, usa um compilador próprio, precisa de Node.js para o ambiente de desenvolvimento e mantém recursos server-side explícitos como `use server`. Isso significa que um preview puramente textual, com transpile JIT no browser, até pode funcionar para demos simples, mas não entrega fidelidade real quando o output começa a depender de imports, aliases, componentes cliente/servidor, ou fluxo de dev server. citeturn2search0turn2search3turn2search4turn2search7turn2search1

O segundo problema é **o uso equivocado de Babel no papel de runtime principal**. A documentação oficial do Babel é bem clara: `@babel/standalone` existe para browser e para casos específicos como prototipação, REPLs e compilação de código fornecido pelo usuário em tempo real, mas, em produção, a recomendação normal é usar um sistema de build em Node.js como Webpack, Rollup ou equivalente. Em outras palavras: usar Babel Standalone como escape hatch é válido; usar Babel Standalone como base da fidelidade de um preview Next.js é frágil por definição. citeturn1search0

O terceiro problema é **a duplicação de custo semântico e temporal**. Os modelos Gemini 2.5 têm janelas de contexto muito amplas, contagem oficial de tokens e mecanismos de caching implícito e explícito. A documentação do Gemini recomenda inclusive contar tokens e aproveitar prefixos comuns para aumentar chance de cache hit. Isso torna a estratégia de “gerar tudo” e depois “reenviar quase tudo para auditar” particularmente ineficiente: você aumenta latência, dobra pontos de falha e ainda desperdiça a oportunidade de estruturar um único passe com regras consistentes. citeturn7search3turn7search0turn7search1

O quarto problema é **dependência inconsistente**. A interface confirmada no repositório usa `window.lucide`, enquanto a documentação pública do Lucide enfatiza consumo por pacotes e tree-shaking. A inferência forte aqui é que o preview atual deve estar misturando dois mundos: geração com semântica de pacote moderno e renderização com expectativa de asset global. Esse tipo de híbrido costuma explodir exatamente na forma que você descreveu: falha de MIME, import quebrado, `srcdoc` silencioso, tela branca e a falsa impressão de que “o modelo parou” ou “faltaram tokens”. fileciteturn30file0L1-L1 citeturn5search0turn3search0turn3search1

O quinto problema, que eu considero plausível mas não pude verificar diretamente nos arquivos exatos citados por você, é o **paradoxo do parser `<preview>`**. Se o sistema ainda tem lógica ativa para capturar HTML paralelo, então mesmo uma instrução forte do tipo “nunca gere `<preview>`” não elimina o risco; ela apenas cria uma superfície de prompt onde o modelo pode “lembrar” da coisa que você quer evitar. Como regra de engenharia, esse tipo de ambiguidade não deve ser resolvido por prompt negativo; deve ser resolvido por **contrato de output**. O modelo só deveria poder devolver um artefato permitido pelo schema.

## Arquitetura recomendada para o Vórtex

A solução que mais faz sentido é uma **arquitetura de artefato único**. O Vórtex não deve mais aceitar dois canais semânticos concorrentes — por exemplo, um bloco de código Next.js e um bloco HTML paralelo. O contrato precisa ser algo como: `framework`, `entrypoint`, `files`, `dependencies`, `previewMode`. Isso pode vir em JSON estruturado ou em uma árvore virtual de arquivos, mas precisa haver **uma fonte da verdade**. Essa mudança, sozinha, já elimina a tentação do parser legar uma “segunda forma” de preview. Ela também aproxima o Vórtex do comportamento do Build mode do AI Studio, que gera arquivos e mostra preview a partir deles, em vez de tentar conciliar linguagens paralelas. citeturn6search3turn6search2

A partir daí, o preview deve ser dividido em **duas pistas de execução**. A primeira é uma pista rápida, barata e conscientemente limitada: componentes React simples, sem `use server`, sem APIs de Node, sem imports complexos, podem continuar usando transpile no browser com Babel ou um serviço ESM moderno. Isso é tolerável porque o próprio Babel reconhece esse caso de uso para protótipos e código do usuário. A segunda pista é a de **alta fidelidade**, acionada quando o artefato pede Next.js de verdade, `app/`, `pages/`, `use server`, rotas, aliases, ou dependências npm relevantes. Nessa pista, o preview deve rodar num runtime Node-capable, preferencialmente em algo da classe dos WebContainers do entity["company","StackBlitz","web ide company"], que executam Node.js e toolchains de build no browser, ou então em um builder efêmero remoto. citeturn1search0turn1search1turn1search9turn2search7turn2search4

Na prática, isso significa abandonar a ideia de que “o navegador vai entender qualquer coisa que pareça Next.js se eu transpirar o suficiente”. O navegador não precisa entender Next.js diretamente; ele precisa receber **um app já servido por um runtime apropriado**. Se vocês quiserem manter tudo client-side, WebContainers é a resposta mais coerente entre as opções pesquisadas, porque foi desenhado justamente para experiências de codificação interativa, execução de Node no browser e apps com execução local do código do usuário. citeturn1search1turn1search9

A auditoria também precisa ser redesenhada. O preview síncrono deve operar em **single pass com validação local**. Em vez de mandar o código de volta ao Gemini para perguntar se ele está conforme, vocês podem fazer três camadas de checagem antes de renderizar: parser AST para garantir integridade estrutural, lint semântico para regras objetivas, e auditoria lexical local para termos banidos ou violações simples. O Gemini continua recebendo as regras no prompt principal, mas a verificação do que é mecânico não pode continuar dependendo de uma segunda rodada síncrona. Se precisarem mesmo de revisão por IA, ela deve ser assíncrona, pós-preview, ou restrita ao fluxo de “publicar”, nunca ao fluxo de “ver a tela”. O próprio repositório mostra hoje um desenho de revisão multipasso; minha recomendação é mantê-lo apenas onde a latência é aceitável. fileciteturn38file0L1-L1 fileciteturn41file0L1-L1 citeturn7search3turn7search0

Por fim, o preview precisa ser **progressivo e desacoplado do término do chat**. O chat termina quando o artefato canônico está válido. O preview começa assim que existe material suficiente para renderização, e continua refinando sem bloquear a UX principal. Esse ponto está em sintonia tanto com o AI Studio quanto com o Firebase Studio, que trabalham com geração visível de arquivos e preview vivo, em vez de amarrar a experiência a um único “momento mágico” no final da resposta. citeturn6search3turn6search6

## Soluções concretas e priorizadas

A ação de maior impacto imediato é **expurgar o parser `<preview>` e qualquer caminho que aceite HTML paralelo**. Não é uma melhoria cosmética; é uma correção de arquitetura. Enquanto existir um segundo canal aceito pelo frontend, o modelo continuará tendo duas formas possíveis de “satisfazer” o sistema, e o preview continuará sendo semanticamente ambíguo. Isso vale mesmo se vocês mantiverem a negativa “nunca gere `<preview>`”; a regra certa é simplesmente não haver parser para esse material.

A segunda ação imediata é **desligar a auditoria síncrona de preview**. O repositório já mostra geração seguida de inspeção Gemini. No Vórtex, essa segunda chamada deve sair do caminho crítico. Se a auditoria é mandatória por compliance, façam duas filas: uma fila de preview, rápida e local; e uma fila de compliance/publicação, mais lenta e revisional. Misturar as duas coisas é o que dá a sensação de preview travado, especialmente quando ainda existe retry/refine automático. fileciteturn38file0L1-L1

A terceira ação é **instrumentar contagem e orçamento de tokens** antes de toda chamada. O Gemini oferece contagem oficial de tokens e metadata de uso; vocês devem registrar, por request, pelo menos: tamanho do prompt, tamanho do contexto fixo, output pedido, tempo de resposta, cache hit, e motivo final de falha. Sem isso, toda discussão sobre “estouro de tokens” vira suposição. Se o problema for timeout, vocês verão timeout. Se for truncamento por parâmetro local, vocês verão truncamento. Se for backpressure ou retry, isso também aparecerá. citeturn7search3turn7search0

A quarta ação é **padronizar a resolução de dependências**. O Vórtex deve escolher explicitamente um dos dois mundos:
- ou o preview é baseado em runtime de pacote, com `package.json`, install virtual e imports reais;
- ou o preview é baseado em browser ESM, com import maps e URLs transformadas de forma explícita.

O que ele não pode continuar fazendo é misturar imports gerados pelo modelo com componentes globais implícitos no `window`. No estado atual, o repositório confirmado já usa `window.lucide`; isso precisa virar política consistente em vez de convenção invisível. fileciteturn30file0L1-L1 citeturn3search0turn3search1turn5search0

A quinta ação é **separar preview “rápido” de preview “fiel”**. React simples, CTA isolado, seção única e componentes sem dependência de servidor podem continuar em fast path. Qualquer coisa que exija Next.js como framework — especialmente App Router, `use server`, roteamento, e build pipeline — deve automaticamente ir para full-fidelity path. Essa linha divisória é importante porque impede a equipe de tentar usar o mesmo motor para duas categorias técnicas muito diferentes. citeturn2search4turn2search7turn1search1turn1search0

## Plano de implementação recomendado

No **curtíssimo prazo**, eu faria uma intervenção de contenção: remover parser `<preview>`, cortar auditoria síncrona no preview, registrar `count_tokens` em todas as chamadas, e corrigir a política de ícones/dependências para que todo o caminho de preview use apenas um método de resolução. Essa etapa não resolve fidelidade total de Next.js, mas deve reduzir drasticamente abortos falsamente atribuídos a tokens. citeturn7search3 fileciteturn30file0L1-L1 fileciteturn38file0L1-L1

No **primeiro sprint estrutural**, eu implementaria o schema de artefato canônico e a classificação automática de preview fast path vs full-fidelity path. Junto disso, criaria um worker local para parsing AST e validações sintáticas/lexicais, de forma que o frontend só tente renderizar código estruturalmente íntegro. A ambição aqui não é “deixar o LLM obedecer melhor”; é “tornar impossível o runtime consumir lixo estrutural”. O Build mode do AI Studio e a prototipação do Firebase Studio mostram que gerar arquivos + preview é um desenho mais robusto que tentar improvisar o preview a partir de texto solto. citeturn6search3turn6search6

No **segundo sprint estrutural**, eu escolheria definitivamente o motor de alta fidelidade. Se a prioridade é permanecer inteiramente no navegador, o candidato natural é WebContainers. Se a prioridade é máxima compatibilidade com Next.js e menor variabilidade entre navegadores, então um builder efêmero remoto pode ser melhor. O critério de escolha deve ser simples: se vocês precisam de App Router e semântica server-side real com previsibilidade, a solução puramente Babel/browser já não serve como motor principal. citeturn1search1turn1search4turn2search1turn2search7

## Veredito sobre a sua proposta Zero-Token Preview Strategy

A proposta é **boa**, mas eu faria três ajustes de formulação.

O primeiro ajuste é que o nome “Zero-Token Preview Strategy” é excelente como slogan, mas tecnicamente o alvo não é zerar tokens; é **zerar tokens redundantes no caminho do preview**. O preview ainda consumirá tokens para gerar o artefato, mas não deve gastar tokens extras com HTML paralelo, auditoria pós-hoc e retries encadeados. fileciteturn38file0L1-L1 citeturn7search0turn7search3

O segundo ajuste é que **expurgar o parser** e **embutir a auditoria** são medidas corretas, porém insuficientes se o runtime continuar incompatível com o artefato gerado. O Vórtex pode até parar de desperdiçar tokens e ainda assim continuar falhando no preview se continuar tentando “fingir” um ambiente Next.js num sandbox que só aguenta React simples. A mudança decisiva é o **motor de preview orientado a framework**. citeturn1search0turn2search4turn2search7turn1search1

O terceiro ajuste é que o “progressive preview” precisa ser tratado como **princípio de UX e de fila**, não apenas como temporização. O preview não deve depender do sucesso da revisão final; ele deve se alimentar do artefato válido mais recente, renderizar o que puder com segurança, e continuar melhorando sem bloquear o chat. Essa separação entre “gerado”, “renderizável” e “publicável” é o que transforma a arquitetura de um hack de chatbot em uma IDE generativa de fato. citeturn6search3turn6search6
