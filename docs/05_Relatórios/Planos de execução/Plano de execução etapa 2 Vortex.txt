# Plano de Execução: Vórtex V7.5.2 — Definitivo

> **Fonte de verdade:** este documento. Tudo anterior é registro histórico.
> **Diretriz técnica:** "Abidos governa. Vórtex executa. Cloudinary provê. CMS preserva. Review protege. Oficina constrói."
> **Estratégia arquitetural:** **Data-driven first.** Toda página é dado versionado renderizado por rota dinâmica única. Code-driven (gerar arquivos `.tsx` em repo) fica documentado no backlog, ativado apenas quando houver caso de uso real fora do schema do AbidosRenderer.
> **Regra de ouro:** uma microetapa de cada vez. Critério de aceite verificado antes de avançar. Sem expansão de escopo durante implementação.

---

## Sumário Executivo

O V7.5.2 está com infraestrutura backend sólida (Postgres/Neon, Cloudinary DAM, Abidos Engine, Review Gate, schema editorial versionado) mas com três pendências estruturais que destravam tudo o resto:

1. **Renderer único** (AbidosRenderer compartilhado entre preview admin e rota pública)
2. **Camada de dados segura** (DAL `server-only` + rota `[[...slug]]` + generateMetadata + notFound)
3. **Contratos formais de Page Ops** (6 tool calls que o chat IA usa para operar páginas)

Após essas três fases, a Oficina Visual Híbrida (Fase 5) tem fundação sólida. Depois entram chat IA com tool calls reais (Fase 6), GERAR + Templates (Fase 7), Release & Observabilidade (Fase 8), Refinamento (Fase 9), Validação E2E (Fase 10).

- [ ] **4.1.2** Implementar `src/lib/vortex/page-ops/readDraft.ts` consumindo DAL
- [ ] **4.1.3** Expor como tool call para Gemini: nome, descrição, parameter `draftId: string`
- [ ] **4.1.4** Cache de 5s para evitar reads repetidos no mesmo turno

### 4.2 — Contrato `mutateSections`

- [ ] **4.2.1** Definir schema Zod do payload: `{ draftId: string, patches: JsonPatch[] }` (RFC 6902)
