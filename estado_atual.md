### **O Caminho para o "Vórtex Pro"**

**Para transformar a ferramenta atual em um produto de nível empresarial, dividi a execução em 4 Sprints (Fases).**

#### Sprint 1: Fluidez e "Vibecoding" Real (Streaming)

**O maior problema atual de UX é a tela de carregamento estática enquanto a IA pensa. Precisamos emular a sensação de "digitação" do DeepSite.**

* **Ação Técnica:** **Refatorar** **lib/ai.ts** **para usar** **generateContentStream** **do SDK do Gemini.**
* **Impacto:** **O código aparecerá no Monaco Editor linha por linha. O Sandpack atualizará o preview quase em tempo real, mantendo o usuário no estado de fluxo (Flow State).**

#### Sprint 2: Arquitetura Multi-Arquivos (O Fim do Monolito)

**Atualmente, a IA cospe todo o código no** **page.tsx**. O DeepSite v4 cria pastas como **components/Hero.tsx** **e as importa.**

* **Ação Técnica:**

  * **Conectar o painel "Explorador" lateral ao banco de dados Dexie (VFS).**
  * **Permitir que a IA retorne um JSON estruturado com múltiplos arquivos (ex:** **[{ path: "components/Hero.tsx", content: "..." }]**).
  * **Injetar todos esses arquivos dinamicamente na propriedade** **files** **do componente** **`<Sandpack />`**.
* **Impacto:** **O Vórtex passa a construir sistemas reais em Next.js, não apenas Landing Pages de uma página só.**

#### Sprint 3: O Guardião Incorruptível (AST Validator)

**O DeepSite confia cegamente na IA. Nós não podemos fazer isso na área da saúde. Se a IA gerar a palavra "Cura", o site não pode ir para o ar.**

* **Ação Técnica:** **Substituir o** **lib/audit.ts** **(que usa Regex) por um** **parser** **real de Árvore de Sintaxe Abstrata (AST), utilizando bibliotecas como Babel ou Acorn rodando em um Web Worker.**
* **Impacto:** **O sistema entenderá a estrutura do React. Ele saberá exatamente se um** **`<h1>`** **está dentro de um** **`<section>`** **ou se uma imagem** **`<Image>`** **do Next.js está sem a propriedade** **alt**, bloqueando o commit automaticamente.

#### Sprint 4: Sincronização e Deploy (Git-in-Browser)

**A ponte final entre o Vórtex e a Vercel.**

* **Ação Técnica:** **Implementar a lógica do** **isomorphic-git** **no painel "Sincronização". Criar a interface onde o usuário revisa o** **diff** **do código, digita a mensagem de commit e clica em "Push to GitHub".**
* **Impacto:** **O Vórtex deixa de ser um "brinquedo de preview" e se torna uma IDE de produção. O código vai para o GitHub, que aciona o webhook da Vercel, publicando o site do Dr. Victor Lawrence em segundos.**

### 🏛️ 1. Arquitetura de Arquivos e VFS (Virtual File System)

**Atualmente, o Vórtex opera quase como um editor de arquivo único (**page.tsx**). Sistemas reais exigem múltiplos arquivos.**

* **Suporte Multi-Arquivos Real:** **O explorador lateral precisa ser funcional. Clicar em** **components/Hero.tsx** **deve carregar esse arquivo no Monaco Editor.**
* **Geração Multi-Arquivos pela IA:** **A IA não pode apenas cuspir código. Ela deve retornar um JSON estruturado (usando** **Structured Outputs** **do Gemini) contendo operações de arquivo. Ex:** **[{ action: "create", path: "components/Button.tsx", content: "..." }, { action: "update", path: "app/page.tsx", content: "..." }]**.
* **Sincronização VFS ↔ Sandpack:** **Toda mudança no banco de dados local (Dexie/IndexedDB) deve ser refletida instantaneamente no objeto** **files** **do Sandpack.**
* **Gestão de Dependências (package.json dinâmico):** **Se a IA sugerir usar** **framer-motion** **ou** **lucide-react**, o sistema deve detectar o **import** **e injetar a dependência automaticamente no Sandpack/WebContainer.**

### 🧠 2. Motor Cognitivo e IA (Gemini 2.5)

**O streaming foi implementado, mas a inteligência contextual precisa de profundidade.**

* **RAG (Retrieval-Augmented Generation) de Repositório:** **A IA precisa conhecer o projeto inteiro. Antes de gerar código, ela deve buscar no VFS quais componentes já existem para reaproveitá-los, evitando recriar botões ou layouts já definidos.**
* **Context Caching:** **Para projetos grandes, enviar todos os arquivos no prompt a cada requisição é caro e lento. Implementar o** **Context Caching** **da API do Gemini para manter a base de código em memória.**
* **Edição Cirúrgica (Diffing/Patching):** **Atualmente, a IA reescreve o arquivo inteiro. Para arquivos de 1000 linhas, isso é ineficiente. A IA deve gerar apenas** **Diffs** **(ex: formato Unified Diff) ou usar ferramentas de** **Search & Replace** **para alterar apenas a função necessária.**
* **Agentes Especializados (MoE local):** **Dividir o prompt monolítico em sub-agentes. Um agente escreve o código, outro agente (Crítico) revisa o código contra as regras do CFP antes de mostrar ao usuário.**

### 🛡️ 3. Governança Clínica e Validação (O Diferencial Abidos)

**O sistema atual usa Regex (**lib/audit.ts**), o que é frágil e fácil de burlar.**

* **AST Validator (Abstract Syntax Tree):** **Implementar um parser real (como** **ts-morph** **ou** **acorn-jsx**) rodando em um Web Worker. Ele deve analisar a árvore do código para garantir que:

  * **Existe exatamente uma tag** **`<h1>`** **renderizada.**
  * **Todas as tags** **`<Image>`** **possuem a propriedade** **alt** **preenchida com texto descritivo.**
  * **Links do WhatsApp possuem os parâmetros UTM corretos.**
* **Auditoria Semântica (NLP Compliance):** **O código gerado deve passar por uma verificação semântica do Gemini para garantir que não há infrações éticas do Conselho Federal de Psicologia (CFP) escondidas no copy (ex: promessas de "cura rápida", "garantia de resultado").**
* **Bloqueio de Commit (Erro Zero):** **Se a auditoria falhar (status vermelho), o botão "Commit & Push" deve ser desabilitado fisicamente na interface.**

### ⚡ 4. Motor de Preview e Compilação

**O Sandpack atual roda um template React básico. O NeuroEngine OS exige Next.js.**

* **Migração para WebContainers (StackBlitz API):** **O Sandpack tem limitações com o Next.js App Router (**app/page.tsx**, Server Components). Integrar a API do WebContainers permite rodar um ambiente Node.js real dentro do navegador, suportando SSR, rotas de API e Server Actions.**
* **Terminal Integrado:** **Adicionar uma aba de terminal real na interface para ver os logs de build do Next.js, erros de linting e instalar pacotes via npm/pnpm.**
* **Web Vitals em Tempo Real:** **Conectar os indicadores do rodapé (LCP, CLS, INP) à API de Performance do navegador dentro do iframe de preview, mostrando métricas reais em vez de dados estáticos.**

### 🔄 5. Versionamento e Deploy (Git-in-Browser)

**Os botões do cabeçalho precisam de vida. O Vórtex deve ser uma ponte direta para a produção.**

* **Integração Isomorphic-Git:** **Usar a biblioteca** **isomorphic-git** **para criar um repositório Git real dentro do IndexedDB.**
* **Autenticação GitHub (OAuth):** **Permitir que o Dr. Lawrence faça login com o GitHub para dar permissão de escrita aos repositórios do Instituto.**
* **Interface de Diff Review:** **Antes de commitar, clicar em "Commit & Push" deve abrir um modal mostrando o "Antes e Depois" (Diff View) do código, permitindo que o usuário aprove as mudanças da IA.**
* **Vercel Webhooks:** **Ao fazer o push para o GitHub, o Vórtex deve escutar o webhook da Vercel e mostrar uma barra de progresso de deploy diretamente na IDE.**

### 🎨 6. UX/UI e Ferramentas de Desenvolvedor

**A interface está bonita, mas precisa de ferramentas de produtividade.**

* **Minimap e Linting no Editor:** **Ativar o minimap do Monaco Editor para arquivos grandes e integrar o ESLint no navegador para sublinhar erros de sintaxe em vermelho enquanto o usuário (ou a IA) digita.**
* **Responsividade Fluida:** **O preview (Mobile/Tablet/Desktop) deve ter uma alça de redimensionamento manual (drag-to-resize) além dos botões fixos.**
* **Histórico de Prompts (Time Travel):** **Salvar o histórico de conversas e permitir que o usuário "volte no tempo" (Ctrl+Z arquitetural), revertendo o código para o estado de 3 prompts atrás.**
* **Gestão de Segredos (Env Vars):** **Um painel seguro para inserir chaves de API (ex: Stripe, SendGrid) que serão injetadas no WebContainer sem serem salvas no código-fonte.**

### 🌐 7. Integração com o Ecossistema NeuroEngine

**O Vórtex não vive isolado; ele é parte de um OS.**

* **Silos & Hubs Sync:** **A aba "Silos & Hubs" deve se conectar a um CMS Headless (ou Firebase) para puxar a arquitetura de conteúdo do Dr. Lawrence. Se ele criar um novo "Silo" de TDAH, o Vórtex deve gerar automaticamente a estrutura de pastas (**app/tdah/page.tsx**).**

* **O "Bottom Drawer" (Gaveta Inferior):** **Remover a Auditoria Abidos e as Regras debaixo do Preview. O Preview precisa de 100% da altura da tela. Mova a Auditoria, Terminal e Logs de Build para um painel inferior colapsável (exatamente como o terminal do VS Code).**

### 🪄 2. Experiência de "Vibecoding" (Interação com a IA)

**A forma como o usuário pede alterações deve ser mágica e fluida.**

* **Inline Diff View (Obrigatório):** **Quando a IA gera código, ela não deve substituir o texto abruptamente. O editor deve mostrar um** **Diff** **(fundo vermelho para o que sai, verde para o que entra). O usuário revisa e clica em "Aceitar" ou "Rejeitar" (atalhos** **Cmd+Enter** **/** **Esc**).
* **Chat Multimodal (Drag & Drop):** **O input do chat deve aceitar imagens. O usuário arrasta um print de um site de referência para dentro do chat, a borda brilha em** **#2dd4bf** **(Teal), e ele digita:** **"Faça o Hero section parecido com isso"**.

* **Animação de "Pensamento" Premium:** **Substituir o texto "Vórtex Pensando..." por um efeito de** **Shimmer** **(brilho percorrendo a borda do input) ou um espectro de áudio sutil, indicando processamento cognitivo profundo.**

### 💎 3. Estética Premium e Glassmorphism

**O** **OLED Black** **está ótimo, mas falta profundidade e refinamento tátil.**

* **Glows Radiais Sutis:** **Adicionar gradientes radiais muito suaves e escuros (ex: um brilho** **#2dd4bf** **com 5% de opacidade) nos cantos da tela ou atrás do editor para quebrar a monotonia do fundo chapado.**
* **Bordas e Sombras Dinâmicas:** **Elementos flutuantes (modais, dropdowns, tooltips) devem ter** **backdrop-blur-2xl**, fundo **bg-white/5** **e uma borda superior de** **border-t-white/10** **para simular o reflexo do vidro.**
* **Scrollbars Customizadas:** **As barras de rolagem nativas do Windows/Mac destroem a imersão. Implementar scrollbars invisíveis que aparecem apenas no** **hover**, finas e com a cor **white/10**.
* **Tipografia Hierárquica:** **Usar** **Inter** **para a UI, mas aplicar** **tracking** **(espaçamento de letras) mais agressivo em uppercase (**tracking-widest**) para** **labels** **e** **badges**, criando um visual mais editorial e técnico.

### 💻 4. Otimização do Editor (Monaco) e Explorador

* **Breadcrumbs:** **Abaixo das abas, mostrar o caminho do arquivo atual (ex:** **neuroengine > app > components > Hero.tsx**).

### 📱 5. Preview e Ferramentas Visuais

**O Sandpack é o coração visual, mas a interação com ele pode ser melhorada.**

* **Resize Fluido (Drag Handle):** **Além dos botões Mobile/Tablet/Desktop, a borda do iframe de preview deve ter um "puxador" (handle). O usuário clica e arrasta para testar a responsividade em larguras arbitrárias (ex: 854px).**
* **Pop-out (Nova Aba):** **Um botão no cabeçalho do preview para abrir a aplicação em uma nova aba do navegador, permitindo testar a tela cheia real sem a UI do Vórtex em volta.**

* **Modo "Foco Clínico" (Zen Mode):** **Um atalho que esconde todas as barras laterais e painéis, deixando apenas o Editor centralizado e o Preview flutuando suavemente ao lado, ideal para sessões longas de escrita de copy terapêutico.**

### 🚀 O Plano de Transformação: Arquitetura Multi-Arquivos

**Para transformar o Vórtex em uma IDE real, precisamos mudar a fundação de como o estado e a IA lidam com o código. Aqui está a lista exaustiva de tudo o que deve ser implementado para uma navegabilidade e gestão de arquivos** **World-Class**:

#### 1. Arquitetura de Dados (Under the Hood)

* **Estado Global de Arquivos (VFS Dinâmico):** **Substituir o** **const [code, setCode]** **por um dicionário de arquivos:** **const [files, setFiles] = useState<Record<string, string>>**.
* **Ponte Sandpack Dinâmica:** **O componente** **`<Sandpack />`** **deve receber esse objeto** **files** **inteiro. Assim, se a IA criar um** **components/Button.tsx** **e importá-lo no** **app/page.tsx**, o Sandpack compilará perfeitamente.
* **O Novo Padrão da IA (XML/Markdown Blocks):** **A IA não pode mais cuspir texto puro. Ela deve ser instruída a usar um formato estruturado que suporte streaming. O padrão ouro para isso é:**

  **code**Markdown

  ```
  <file path="components/Hero.tsx" action="create">
  export function Hero() { ... }
  </file>
  <file path="app/page.tsx" action="update">
  import { Hero } from '@/components/Hero';
  ...
  </file>
  ```

  **O frontend fará o** **parse** **dessas tags em tempo real durante o streaming, criando e atualizando as abas correspondentes.**

#### 2. Navegabilidade e UX do Explorador (File Tree)

* **Árvore de Diretórios Real (Collapsible Tree):** **O explorador lateral deve renderizar pastas reais que podem ser expandidas/colapsadas (ex:** **app/**, **components/**, **lib/**).
* **Abas do Editor (Editor Tabs):** **O topo do Monaco Editor deve ter abas horizontais para os arquivos abertos. Clicar em um arquivo no explorador abre uma nova aba.**
* **Indicador de "Não Salvo" (Dirty State):** **Se o usuário (ou a IA) alterar um arquivo, a aba e o explorador devem mostrar uma bolinha branca (•) indicando que há mudanças não salvas no VFS/Dexie.**
* **Quick Open (Cmd/Ctrl + P):** **Implementar uma paleta de comandos flutuante. O usuário digita** **Cmd+P**, começa a digitar "Hero" e o sistema filtra os arquivos, permitindo pular para o código instantaneamente sem usar o mouse.

#### 3. Interação da IA com Múltiplos Arquivos (Context Awareness)

* **@Mentions no Chat:** **O usuário deve poder digitar** **@** **no chat para abrir um menu suspenso com a lista de arquivos. Ex:** **"Refatore o @components/Header.tsx para usar a cor primária definida no @tailwind.config.ts"**.
* **Auto-Contexto Inteligente:** **Se o usuário não mencionar nenhum arquivo, a IA deve receber automaticamente no prompt o arquivo que está** **atualmente aberto e visível** **no editor, assumindo que a intenção do usuário é alterá-lo.**
* **RAG de Repositório (Busca Semântica):** **Se o usuário pedir "Crie uma página de contato", a IA deve primeiro buscar no VFS se já existe um** **components/Form.tsx** **para reaproveitá-lo, em vez de recriar um formulário do zero.**

#### 4. Gestão de Rascunhos (Drafts & Staging)

**Como estamos lidando com código clínico sensível, a IA não deve sobrescrever arquivos críticos sem supervisão.**

* **Modo "Review" (Diff Editor):** **Quando a IA propõe mudanças em múltiplos arquivos, os arquivos afetados devem ficar com a cor roxa/azul no explorador (indicando "Modificado pela IA").**
* **Aceitar / Rejeitar por Arquivo:** **O usuário clica no arquivo modificado, vê o** **Diff** **(lado a lado, o que era vs. o que a IA fez) e pode clicar em "Aceitar Alteração" ou "Descartar".**
* **Histórico de Versões Local (Snapshots):** **Antes de cada geração da IA, o sistema tira um "Snapshot" (foto) de todo o VFS. Se a IA quebrar o site, o usuário clica em "Desfazer Última Ação da IA" e todos os arquivos retornam ao estado anterior.**

#### 5. Operações Manuais de Arquivo (CRUD)

**O usuário (Dr. Lawrence ou desenvolvedor) também precisa de controle manual.**

* **Menu de Contexto (Right-Click):** **Clicar com o botão direito no explorador deve abrir um menu:** **Novo Arquivo, Nova Pasta, Renomear, Excluir**.
* **Drag & Drop:** **Permitir arrastar arquivos entre pastas no explorador lateral para reorganizar a arquitetura.**
